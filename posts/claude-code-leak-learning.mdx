# Claude Code 源码泄露事件学习笔记
> 事件：Claude Code 2.1.88 源码意外泄露
> 类型：AI 工程架构、安全、发布事故学习

## 事件概述
- 泄露原因：Anthropic 工程师在发布 Claude Code 2.1.88 时，打包工具 Bun 默认生成的 **Source Map 文件（cli.js.map）** 被一起打进了 npm 发布包。
> 通俗解释：就像你寄快递，不小心把**产品设计图纸、组装说明书**也一起塞进去寄给了用户，别人拿到就能直接复刻一摸一样的东西。

- 泄露内容：包含 **1906 个 TypeScript 源文件**，约 **51.2 万行代码**。
> 通俗解释：等于把整个 Claude Code 的**完整配方、内部结构、运行逻辑**全部公开了。

- 发现者：安全研究员 Chaofan Shou 在 X（原 Twitter）上最先曝光。
> 通俗解释：最早是安全圈的人在使用时发现不对劲，然后公开了这件事。

- 泄露方式：`.map` 文件是 JSON 格式，里面有 `sources`（文件路径）和 `sourcesContent`（完整源码）两个关键数组。
> 通俗解释：这个文件就是一个**“代码还原对照表”**，一边是文件名，一边是源代码，一对照就能把整个项目复原。

## 架构设计亮点

### 1. Agent 循环引擎
- 核心机制：`while(true)` 无限循环实现 **ReAct（推理+执行）** 模式。
> 通俗解释：让 AI 像人一样干活：**想一步 → 做一步 → 看结果 → 再想下一步**。

- 工作流程：
1. 上下文压缩
2. 调用大模型获取流式响应
3. 解析工具调用请求
4. 执行工具
5. 结果追加到对话历史
> 通俗解释：AI 的标准工作流：先整理思路 → 问大脑下一步干嘛 → 解析指令 → 动手做事 → 把结果记下来，循环往复。

- 模型调用：通过 `services/api/claude.ts` 中的 `queryModel` 函数。
> 通俗解释：这是 AI 和“大脑”（大模型）通信的统一入口，所有思考都走这里。

---

### 2. 工具系统设计
- 40+ 内置工具：通过 `getAllBaseTools()` 统一注册。
> 通俗解释：AI 会的所有技能（读文件、写代码、搜索、执行命令）都在这里统一登记。

- fail-closed 原则：工具默认不安全（`isConcurrencySafe: false, isReadOnly: false`），必须显式声明才开放。
> 通俗解释：**默认不信任所有工具**，除非开发者明确说“这个功能是安全的”，否则不让用。

- 动态加载：工具多的时候，先给精简列表，用到再加载完整定义。
> 通俗解释：像餐厅菜单，先给菜名，你点了再上详细做法。

- 并发控制：
- 默认最大 10 个并发工具
- 只读工具可并行，写操作必须排队
- 策略：读读并行、读写互斥
> 通俗解释：很多人可以一起看书，但有人要修改时，必须单独来，不然内容会乱。

---

### 3. 三层记忆架构
1. **MEMORY.md（热数据）**
- 常驻内存，限制 200 行/25KB
- 截断时按换行符切分，不乱切结构
- 超出会加 WARNING 提示
> 通俗解释：AI 的**随身小纸条**，只能记一点点，记不下就提醒你。

2. **话题文件（温数据）**
- 按需加载（最多加载 5 个相关文件）
- 用小模型挑选相关记忆
- 不存代码，避免缓存不一致
> 通俗解释：像书柜，聊到什么主题就拿什么书，不存会变的代码。

3. **历史对话（冷数据）**
- 存成 JSONL 格式
- 用 grep 搜索检索
> 通俗解释：旧记录存仓库，要用时搜一下就行。

---

### 4. 五级上下文压缩
1. Snip 剪裁：旧工具结果只保留结构
2. Microcompact 微压缩：大内容移到缓存
3. Context Collapse 折叠：中间对话自动摘要
4. Autocompact 自动压缩：超量就触发
5. Reactive Compact 应急压缩：API 返回 413 时强制清理

> 通俗解释：像手机空间不足时的清理策略：
> 先删不重要的 → 再删大文件 → 合并内容 → 实在不够就紧急清理。

- 断路器机制：连续失败 3 次后停止尝试。
> 通俗解释：像保险丝，连续出错就自动断电，避免无限浪费钱和资源。

---

### 5. 系统提示词缓存
- 通过 `DYNAMIC_BOUNDARY` 分割提示词
- 静态部分：全球用户共享缓存
- 动态部分：个性化内容单独加载
- 效果：大幅节省 token 成本

> 通俗解释：就像发通知，**固定抬头大家共用，个人信息单独填**，又省钱又高效。

---

### 6. 安全设计
- YOLO 模式审查：就算开了危险模式，仍有影子 AI 分类器把关。
> 通俗解释：你说“随便搞”，背后依然有个 AI 保镖盯着，真危险了会拦着。

- Bash 安全检测：23 种安全检查规则。
> 通俗解释：有 23 条规则检查你让 AI 执行的命令是不是恶意的。

- 多层权限过滤：
1. 运行模式检查
2. 用户自定义钩子
3. YOLO 分类器
4. Bash 危险命令拦截
5. 配置规则引擎
> 通俗解释：过 **5 道安检**，任何一道报警都不让执行。

- 反蒸馏机制：向 API 请求注入假工具定义。
> 通俗解释：给想偷技术的人喂假数据，让他们学歪。

- 卧底模式：内部员工给开源项目贡献代码时，自动隐藏 AI 痕迹。
> 通俗解释：用 AI 写开源代码时，不会暴露是 AI 生成的。

## 工程技术细节

### 1. 性能优化
- 懒加载：入口文件 `cli.tsx` 动态导入模块。
> 通俗解释：用到什么功能才加载什么，不一次性全读进内存。

- 早期输入捕获：启动期间缓存用户输入。
> 通俗解释：软件还没完全开好，你打的字也能存下来，开好立刻用。

- TCP 预连接：启动时提前连接 API，节省 100~200ms。
> 通俗解释：一边开机，一边提前和服务器打招呼，加快速度。

---

### 2. Feature Flag 管理
- 实验功能通过条件 `require` 引入
- 支持按用户/环境灰度发布
- 未开启的功能会被打包工具剔除

> 通俗解释：功能开关，想给谁用就给谁用，不用的功能直接删掉，不占体积。

---

### 3. 隐藏功能曝光
- KAIROS 模式：长期运行助手 + AutoDream（夜间自动整理记忆）
> AI 24 小时在线，晚上还会“做梦”整理白天的内容。

- COORDINATOR_MODE：多 Agent 协作（研究→综合→实现→验证 四阶段）
> 一个 AI 总指挥，多个小弟 AI 分工干活。

- VOICE_MODE：语音交互
> 可以直接语音和 AI 对话。

- WEB_BROWSER_TOOL：浏览器操作工具
> AI 能自己打开浏览器、搜网页、点按钮。

- 数字宠物系统：18 种虚拟宠物（鸭子、仙人掌等）
> 写代码累了可以撸 AI 宠物解压。

---

## 可借鉴的设计思想
1. **简单有效胜于复杂**：用 grep 搜索替代复杂 RAG，模型强时更稳定。
2. **分层管理**：热/温/冷数据用不同策略。
3. **fail-closed 安全原则**：不确定安全 → 默认禁止。
4. **渐进式压缩**：从轻到重逐级清理。
5. **边界清晰**：工程系统与 AI 模型职责分开。

---

## 工程实践启示
1. 发布流程：生产环境必须过滤 `.map` 文件。
2. 缓存设计：公共内容与个性化内容分离。
3. 错误处理：必须加断路器，防止雪崩。
4. 性能优化：预连接、懒加载、输入缓冲组合使用。
5. 安全防御：多层防护 + 主动反制策略。
